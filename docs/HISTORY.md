# History

`HISTORY.md`는 요구사항과 의사결정 맥락의 단일 소스다. 단순 변경 목록은 `docs/CHANGELOG.md`에 기록하고, 이 문서는 왜 그런 요구사항이나 결정이 생겼는지를 시간순으로 남긴다.

## 2026-04-18

### 초기 제품 방향 확정

- Python `FastAPI` + `APScheduler` + `SQLite` 백엔드와 Galaxy S24용 `React/Vite PWA`로 구축하기로 했다.
- 초기 검증은 Windows/WSL 로컬 환경에서 수행하고, 안정 운영 전환 시 클라우드/VPS와 PostgreSQL로 이전한다.
- v1은 국내+미국 주식만 다루며 코인/선물/옵션은 제외한다.
- v1은 자동 매매 주문을 실행하지 않고 투자 판단 보조 리포트와 알림에 집중한다.

### 관심종목과 보유종목 분리

- 관심종목은 매수 후보, 관찰 대상, 뉴스/가격 급변 감시 중심으로 관리한다.
- 보유종목은 실제 보유 중인 종목으로 수량, 평균매수가, 매수일, 목표가, 손절가, 메모를 수동 입력한다.
- 같은 ticker가 관심종목과 보유종목에 동시에 있을 수 있게 한다.
- 보유종목 리포트는 매도/추가매수/보유 판단을 더 강하게 다루고, 관심종목 리포트는 진입 조건과 매수 대기 기준을 중점으로 다룬다.

### Codex CLI 분석 방향

- Codex CLI는 `/usr/bin/codex`를 사용한다.
- 정기 분석과 경제뉴스는 균형형 multi-agent pipeline을 사용한다.
- 5분 급변 알림은 규칙 기반 판정을 우선하고, 큰 이벤트나 뉴스가 붙은 경우에만 가벼운 단일 Codex 요약을 붙인다.
- Codex 실패/timeout 시에도 기본 규칙 기반 알림과 원자료 링크는 발송한다.

### 실행 가능한 scaffold 구현

- 빈 저장소에서 문서만 있는 상태였기 때문에, 우선 실제 API와 PWA가 실행되는 최소 골격을 구현했다.
- 외부 provider(KIS, 뉴스/공시/SNS, 실제 Codex CLI, FCM, SMTP)는 dry-run/mock 경계로 남겨 다음 단계에서 교체할 수 있게 했다.
- 자연어 명령은 명확한 관심종목/보유종목 등록을 즉시 실행하고, 전문가 소스는 URL이 없으면 확인을 요청하되 오건영 Facebook처럼 후보 URL이 있는 소스는 비활성으로 등록하기로 했다.
- APScheduler는 FastAPI lifespan에서 시작하는 구조로 두고, 실제 도메인 job 구현 전 heartbeat scaffold만 등록했다.

### 라벨형 자연어 명령 처리

- 사용자가 `보유종목: 삼성전자`처럼 짧은 라벨형 명령을 입력할 수 있음이 확인됐다.
- 해당 표현은 미지원으로 처리하지 않고 보유종목 의도로 인식하되, 수량과 평균매수가가 없으므로 실행하지 않고 추가 입력을 요청하도록 정했다.
- `관심종목: 삼성전자`도 즉시 등록하지 않고 `추가` 또는 `등록` 의사를 확인하도록 했다.

### API Orchestrator 방향 확정

- `/api/commands`는 단순 키워드 파서가 아니라 Codex skill 스타일의 API orchestrator 역할을 해야 한다.
- 사용자의 자연어를 지원 기능과 API action에 매핑하고, 실행 가능한 요청은 바로 실행한다.
- 지원 기능이지만 필수 정보가 부족하거나 애매하면 guide를 반환해 제대로 요청하게 돕는다.
- 지원하지 않는 기능은 기능이 없다고 명확히 반환한다.
- 서버 코드에서 문자열 parser/정규식/종목 하드코딩으로 자연어를 해석하지 않고, `api-orchestrator` skill과 Codex JSON action plan을 사용하기로 정했다.
- 자연어 명령은 패턴 우회 없이 Codex orchestrator가 action/slots를 만들고, backend는 그 결과만 실행한다.

### 자연스러운 보유 표현 지원

- `삼성전자 160500원 284주 보유`처럼 UI 용어인 `보유종목`을 쓰지 않아도 보유종목 등록 의도로 처리하기로 했다.
- 가격 단위가 붙은 숫자는 평균매수가로, `주` 단위가 붙은 숫자는 수량으로 해석한다.
- 같은 `ticker+market` 보유종목은 여러 row로 중복 저장하지 않고 현재 보유 상태를 업데이트한다.

### 대시보드 표시와 수동 테스트 요구사항

- 보유종목 카드에는 현재주가, 수량, 갯수를 표시하고 값이 없으면 `-`로 표시한다.
- 관심종목 카드에는 관심종목명과 현재주가를 표시한다.
- 기본 스케줄은 처음 실행 시 seed로 채우고, 사용자가 클릭하면 팝업으로 상세 내용을 확인할 수 있게 한다.
- 분석 리포트와 경제뉴스 소스도 비어 보이지 않도록 초기 seed 데이터를 둔다.
- 각 기능은 스케줄 실행을 기다리지 않고 PWA 버튼으로 dry-run 테스트할 수 있게 한다.
- 분석 리포트는 목록에서 선택하면 전체 화면에서 Markdown 본문을 자세히 읽을 수 있게 한다.
- 스케줄 목록의 각 항목은 사용자가 기다리지 않고 즉시 실행할 수 있어야 한다.
- 수동 스케줄 실행은 dry-run이 아니라 현재 연결된 실제 provider(KIS, Telegram)를 사용해야 한다.
- 분석/뉴스 스케줄 실행 결과는 템플릿이 아니라 Codex가 주어진 context를 분석해 작성한 Markdown이어야 한다.
- 스케줄은 타입별 전문 skill을 사용해 일관된 리포트 구조를 유지하고, 리포트 생성 완료/실패도 반드시 알림으로 전달한다.
- Oracle Cloud 배포는 Docker Compose와 PostgreSQL을 기본 실행 형태로 삼고, 경제뉴스 소스는 사용자가 UI에서 언제든 추가/삭제할 수 있어야 한다.
- 관심분야는 관심종목과 별도 category로 관리하고, 매일 09:00 KST에 연구성과가 연결 종목 전망에 의미 있게 이어지는지 감지해 주요 성과가 있으면 알림을 보내기로 했다.
- 글로벌 경제뉴스는 소스 확인 목록이 아니라 실제 수집 헤드라인을 바탕으로 시장 영향과 최적의 투자 액션을 전달해야 한다.
- Oracle/Docker 운영에서도 Codex 리포트가 동작해야 하므로 사용자가 `secrets/codex/auth.json`을 직접 저장하고 backend 컨테이너가 이를 `/root/.codex/auth.json`로 읽도록 구성한다.
- WSL Docker bridge에서 외부 HTTPS가 timeout나면 Codex orchestrator도 동작하지 않으므로, 로컬 WSL override는 backend를 host network로 실행해 Codex CLI 연결성을 우선 보장한다.
- 경제뉴스 소스도 사용자가 내부 필드를 직접 관리하지 않고 자연어 요청으로 추가/삭제하며, category/platform/enabled 같은 구조화는 orchestrator와 backend가 맡는다.
- 사용자는 여러 줄 포트폴리오를 한 번에 붙여넣을 수 있어야 하므로, 자연어 명령은 단일 action뿐 아니라 여러 action을 담은 batch plan도 실행한다.
- Codex가 ETF ticker를 일부 잘못 추정할 수 있으므로, backend executor는 확인된 주요 보유 종목명을 기준으로 ticker를 보정한다.
- Oracle Cloud는 계정만 생성된 상태이므로, 먼저 GitHub Actions CI와 수동 CD workflow를 구성하고 VM/SSH secret 준비 후 버튼 배포로 전환한다.
- 1GB/1core 무료 VM에서는 PostgreSQL보다 SQLite 파일 DB가 적합하므로, Oracle 기본 배포는 lite compose로 시작하고 필요 시 PostgreSQL로 전환한다.

### 한국투자증권 API 환경 설정

- 한국투자증권 Open API는 실전과 모의 환경을 모두 설정하되, 실행 환경은 `.env`의 `KIS_ENV`로 전환한다.
- App Key와 App Secret은 코드에 저장하지 않고 `.env`에 사용자가 직접 입력한다.
- 토큰 발급은 선택된 환경의 base URL과 `client_credentials` grant type을 사용한다.
- 접근토큰은 1분당 1회 발급 제한이 있으므로 만료 전까지 재사용하고, 토큰 오류가 확인될 때만 1회 재발급한다.
- 여러 국내 종목 현재가는 단건 현재가 API 반복보다 한국투자증권 `관심종목(멀티종목) 시세조회`를 우선 사용하고, API 한도인 30종목 단위로 나누어 처리한다.
